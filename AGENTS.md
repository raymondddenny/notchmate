## Build

- Native macOS app, Swift + SwiftUI, deployment target macOS 14. Hand-authored `.xcodeproj` (no CocoaPods/Carthage/SPM deps; zero third-party).
- Headless build (two steps - signing is required for Spotify TCC):
  1. `xcodebuild -project notchmate.xcodeproj -scheme notchmate -configuration Release -derivedDataPath build CODE_SIGNING_ALLOWED=NO build`
  2. `codesign --force --deep --options runtime --entitlements notchmate/notchmate.entitlements --sign - build/Build/Products/Release/notchmate.app`
  Output: `build/Build/Products/Release/notchmate.app`.
  The xcodeproj sets `CODE_SIGN_IDENTITY = "-"` and `CODE_SIGN_ENTITLEMENTS = notchmate/notchmate.entitlements` for Xcode IDE builds; the CLI path needs the explicit post-build `codesign` step because `CODE_SIGNING_ALLOWED=NO` overrides it.
  **`--options runtime` is now required**: on macOS 26 (26.5.1, confirmed 2026-06-25), the OS blocks outgoing Apple Events for apps without hardened runtime + the `com.apple.security.automation.apple-events` entitlement. Without these flags the app never appears in System Settings > Privacy > Automation and the Spotify widget is permanently stuck on "Nothing playing". Plain `--sign -` (without `--options runtime`) no longer suffices on macOS 26.
- `Info.plist` is checked in and wired via `INFOPLIST_FILE` with `GENERATE_INFOPLIST_FILE = NO`. `LSUIElement = true` (no dock icon / agent app), so the app has no standard window - all UI is the NSPanel.
- **Adding a Swift file (hand-authored pbxproj) - IDs MUST be unique:** wire each new file in 4 places (PBXBuildFile, PBXFileReference, the `notchmate` PBXGroup children, PBXSourcesBuildPhase). The object IDs are 24-char **hex** strings. **Do NOT reuse an existing ID**: the `DCDC0000...A4`/`B4` pair, for example, is already taken by `ducky_2_spritesheet.png`. If a new source reuses an in-use ID, xcodebuild silently drops one of the entries and the file never compiles - the symptom is a baffling `cannot find 'TypeName' in scope` with no compile error in the new file itself. Pick fresh hex suffixes (e.g. `...E1`/`...E2`) and confirm they're unused first.

## Architecture

Free-tier, no license gate.
- `HUDs/` - `HUDController` (ObservableObject; volume observed via `AudioObjectAddPropertyListenerBlock` on the default output device - no permission needed; brightness polled 0.1s via `DisplayServicesGetBrightness` private framework loaded with dlopen; publishes `HUDEvent?` enum that auto-nils after 1.5s; observes `NotchPreferences.$hudSuppressSystem` to bootout/bootstrap `com.apple.OSDUIHelper`; restores OSDUIHelper on `NSApplication.willTerminateNotification`; all listeners removed in `deinit`), `HUDView` (icon + level-bar + percentage HStack), and `HUDOverlayView` (wraps HUDView with dark pill background; the content of the HUD overlay panel below the notch). Free-tier, no license gate.
- **HUD settings structure (master + sub-toggles):** `hudSuppressSystem` is the master toggle in Settings > HUDs. `hudVolumeEnabled` and `hudBrightnessEnabled` are sub-toggles that are only shown in the UI when master is ON. When master is turned ON (the `didSet` in `NotchPreferences`), both sub-toggles are reset to `true` automatically. `HUDController.handleVolumeChange` and `checkBrightness` both check `hudSuppressSystem && hudVolumeEnabled/hudBrightnessEnabled`, so events are suppressed when master is OFF regardless of sub-toggle state.
- **HUD below-notch panel (slide+fade):** `NotchWindowController` owns a second `hudPanel: NSPanel` (borderless, non-activating, `ignoresMouseEvents = true`, `alphaValue = 0` default). Its content is `HUDOverlayView(hud: hud)`. The `hud.$currentEvent` publisher is mapped to a `Bool` (present/absent) and `.removeDuplicates()`d, so only the visibility *transition* drives the animation - consecutive value changes (volume 50→60) just re-render the bar in place via the observed `HUDController`, no re-slide. `setHUDPanel(visible:)` slides the pill: on show it seeds the hidden frame (raised `hudSlideOffset=12pt` toward the notch, `alpha=0`) then animates down to `hudRestingFrame()` + `alpha=1` (0.22s easeOut); on hide it slides back up + fades (0.32s easeIn). The resting frame is `gap=6pt` below the collapsed panel's bottom edge (computed from `geometry.collapsedSize` to stay stable during expand/collapse). Visible window is ~1.5s (the `HUDController` auto-nils `currentEvent`). HUD beats the lyrics strip: while visible it sets `hudActive=true` which hides the lyrics strip. `NotchView` does not contain `HUDView` in its collapsed strip.
- **Lyrics below-notch strip:** `NotchWindowController` owns a third `lyricsPanel: NSPanel` (same setup as hudPanel: 340x44pt, borderless, non-activating, ignores mouse). Its content is `LyricsStripOverlayView(lyrics: lyrics)`. `updateLyricsPanel()` shows it (0.2s) when: synced lyric line is present AND panel is not expanded AND HUD is not active; hides it (0.3s) otherwise. `positionLyricsPanel()` places it at the same slot as the HUD panel (gap=6pt below collapsed bottom); called once in `show()`. `isExpanded`, `lyrics.$state`, and `hud.$currentEvent` all trigger `updateLyricsPanel()`. HUD takes visual priority - when both would show, the HUD wins by setting `hudActive = true` which suppresses the lyrics strip.

A present-but-unparseable settings.json aborts (never clobbered). `#if DEBUG runSelfCheck()` proves the merge/unmerge is idempotent + reversible.

Focused repo = `pickDir` = the dir of the highest-pid session (newest ≈ most-recently-active). For this `ClaudeSession` carries the full `dir` (not just the basename `project`). Pure helpers (`pickDir`, `parseAheadBehind`) have a `runSelfCheck()`. Poll 4s + re-target on session change.

- **Expanded panel sizing (responsive):** `NotchWindowController.currentExpandedSize` uses `geometry.expandedSize(moduleCount:rowCount:)` where `moduleCount = prefs.visibleModules.count` (capped at 3) and rowCount comes from `NotchPreferences.expandedRowCount`. The grid splits modules into `rows = min(rowCount, count)` rows, so the widest row has `cols = ceil(count/rows)` columns. **Width and height both derive from content**: `width = cols*268 + gaps + 2*padH` (each tile gets a fixed comfortable 268pt column so titles/artist don't truncate), `height = notchHeight + rows*118 + gaps + padV` (118pt row fits the media tile's artwork+lyric+controls). Floored at `notchWidth+220`. Panel re-fits when `enabledModules`, `moduleOrder`, or `expandedRowCount` changes (merged publisher).
- **Collapsed strip composition:** mascot (if enabled) is pinned to the **left edge**; equal `Spacer`s on both sides keep the chip cluster centered under the notch. Chips: media controls (prev/play/next only - no artwork, no text) | timer countdown (if running) | git branch (if repo) | CPU% (if above threshold). Claude sessions are expanded-only (kept out of the collapsed strip to stay lean). Current lyric shows in the below-notch strip panel, not inside the collapsed strip. Each chip self-hides until it has signal.
- **Mascot mood derivation:** `MochiMood.derive(spotifyPlaying:spotifyPresent:claudeSessions:)` is a pure function (unit-testable; `runDeriveSelfCheck()` asserts the truth table from the SwiftUI preview). Priority: `dancing` (Spotify playing) > `thinking` (>=1 Claude session) > `idle` (a track loaded but paused) > `sleeping` (nothing). Dancing deliberately wins over thinking. **Extension point:** the app only knows session *count*, not working-vs-waiting; when a finer signal lands, branch inside `derive` (a "working" session -> livelier `thinking`, "waiting" -> calmer pose) and nothing in the view changes.
- **Mascot rendering:** pure SwiftUI shapes/`Path` + a `TimelineView(.animation)` clock - zero assets, zero deps. Continuous wobble/breathe/glow/blink come from `MochiPose.at(t,for:)` (bounded sin curves so a mood switch never jumps). Discrete face features (eye/mouth styles, cheeks, accent) crossfade via animatable opacities driven off an `@State mood` updated in `withAnimation`, so mood changes glide rather than snap. Reduce Motion (`accessibilityReduceMotion`) drops the per-frame `TimelineView` for a still pose.
- **Claude status lights - hook-driven, NOT a process scan.** Live state (running/waiting/idle) is impossible to derive from a `ps` scan, so each Claude Code session reports its own state via hooks:
  - **Status files:** `~/.notchmate/sessions/<session_id>.json` = `{ state, name, project, branch, cwd, pid, updated }`. `state` ∈ `running|waiting|idle`; `updated` is epoch seconds; `name` is the session discriminator (see below). `ClaudeSessionsController` reads these every 2s and drops any whose `pid` is dead or whose file is >6h stale (deleting the orphan).
  - **Helper:** `~/.notchmate/bin/notchmate-hook <state>` reads the hook JSON on stdin, extracts `session_id`/`cwd` with `/usr/bin/plutil -extract <key> raw -o - -` (zero-dep, ships on all macOS - no jq/python), computes `project`=basename(cwd) and `branch`=`git -C cwd rev-parse --abbrev-ref HEAD`, resolves the owning `claude` pid by walking up the process tree from `$PPID`, and writes the file atomically (tmp+`mv`). Fail-silent (always `exit 0`) so it never blocks a turn.
  - **Event → state mapping** (written into `~/.claude/settings.json` by `ClaudeHookInstaller`, no `matcher` so each matches all): `SessionStart`→`idle` (register), `UserPromptSubmit`→`running`, `PreToolUse`→`running` (re-yellow after a permission pause), `Notification`→`waiting` (needs confirmation/input), `Stop`→`idle` (turn done), `SessionEnd`→`end` (removes the file).
  - **Session discriminator (`name`):** firstmate crewmates all share cwd basename `firstmate` + branch `main`, so project+branch alone collapse to identical labels. The helper picks `basename($FM_HOME)` (e.g. `notchy`/`wedding-dashboard`), else the tmux session name (`tmux display-message -p '#S'`), else the pid. The widget shows `displayName` = that discriminator when meaningful, else `project`.
  - **Schema/format confirmed from current docs** (https://code.claude.com/docs/en/hooks.md): hook JSON arrives on **stdin** with `session_id`, `cwd`, `hook_event_name`, `transcript_path`; settings format is `{"hooks": {"<Event>": [{"hooks": [{"type":"command","command":"..."}]}]}}`; `UserPromptSubmit`/`Stop` take no matcher.
- **Notch detection:** `screen.safeAreaInsets.top > 0` => notch present; notch width = `frame.width - auxiliaryTopLeftArea.width - auxiliaryTopRightArea.width`. No notch => top-center pill (all corners rounded, small top gap).
- **Premium seam:** each feature is a self-contained widget folder. Future premium widgets (notifications, fleet view, themes) get their own folder and are gated behind a license check at the point they're added to `NotchView` - the notch shell stays feature-agnostic. No paywall exists yet.
- **Design tokens - `Theme.swift`:** `notchmate/Theme.swift` centralises the spacing scale (4-pt grid: `sp1`..`sp5`), typography roles (`labelFont`, `primaryFont`, `secondaryFont`, `captionFont`, `chipFont`), shared text-color constants (`textPrimary`..`textDisabled`), per-widget accent colors (`accentTimer`, `accentClaude`, `accentGit`, `accentCPU`, `accentMem`), shared fills (`trackBackground`, `dividerColor`), panel surface colors (`panelBackground` = `#15171A` dark cool charcoal, `panelSurface` = `#1C2028` elevated tile surface, `panelBorder`), and tile layout constants (`tileGap`, `tileCorner`). The background is NOT pure black - use `Theme.panelBackground` everywhere, never `Color.black`. Every widget imports these instead of using magic numbers.
- **Default media source is Spotify (Web API) only:** `NotchPreferences.init` always sets `mediaSource` to `.spotifyWeb`; legacy `.spotify` (AppleScript) is migrated away. `MediaPane` no longer shows a source picker - only Spotify Web API is active; Apple Music and YouTube Music are shown as "coming soon" placeholders. The `MediaSource` enum retains all three cases for controller logic but the UI hides them.
- **Horizontal layout model:** The expanded panel renders widgets as module tiles in a configurable horizontal grid (1-3 rows). Key components:
  - `LayoutModule` enum (in `NotchPreferences.swift`): `.media`, `.mochi`, `.timer`, `.git`, `.claude`, `.stats` - each with `displayName`, `summary`, and `icon`. (There is no `.lyrics` case - lyrics live inside the media tile.)
  - **Max 3 visible modules:** `NotchPreferences.maxVisibleModules = 3`. `visibleModules` = `Array(orderedEnabledModules.prefix(3))` is the single source of truth for what renders (panel + geometry). `canEnableMoreModules` (`enabledModules.count < 3`) and `toggleModule` enforce the cap on the way in; `prefix(3)` is a defensive backstop on the way out.
  - `NotchPreferences` holds `moduleOrder: [LayoutModule]`, `enabledModules: Set<LayoutModule>`, `expandedRowCount: Int` (1-3). All persisted to `UserDefaults` as raw-value string arrays/int. `orderedEnabledModules` is `moduleOrder.filter { enabledModules.contains($0) }`.
  - Default enabled: `.media`, `.claude`, `.timer` (3 modules, 2 rows). Default order: media, claude, timer, mochi, git, stats.
  - `NotchView.chunkedRows(_:rowCount:)`: splits `visibleModules` into N sequential chunks (ceil division). With 3 modules and 2 rows: row 1 = [media, claude], row 2 = [timer].
  - Each tile is `frame(maxWidth: .infinity, maxHeight: .infinity)` with `Theme.panelSurface` background rounded 10pt, so tiles fill their grid cell. Content is the widget's `expanded: true` view.
  - Panel dimensions are responsive (see "Expanded panel sizing"): width = `cols*268` + gaps + padding, height = `rows*118` + gaps + padding. Panel re-fits when `enabledModules`, `moduleOrder`, or `expandedRowCount` changes.
  - `showLyrics` (gates the `LyricsController` fetch/poll lifecycle) is derived from `enabledModules.contains(.media)` - lyrics show whenever the media module is on. Synced in `toggleModule(.media)`. Never set independently.
  - Settings > Layout pane (`LayoutPane.swift`): row count radio picker, module list (name + summary) with enable toggles + drag-to-reorder (`ForEach.onMove`), a "N of 3 shown" counter, toggles disabled once the cap is reached, Reset to Defaults button.
- **`NotchPreferences.init` two-phase pattern:** Uses `_property = Published(wrappedValue: ...)` (direct backing store init) for ALL stored properties before any UserDefaults reads, so `self` is fully initialized before Phase 2. This avoids the Swift "self used before all stored properties initialized" error that occurs when `didSet` closures reference cross-properties. Phase 1 = backing init (no didSet, no UserDefaults writes). Phase 2 would use setters, but instead all UserDefaults reads are pre-computed into local `let` constants and passed to the backing stores directly. `didSet` observers run on user-triggered mutations only (post-init).

## Sharp edges

- **Hover-region oscillation trap:** `.onHover` (backed by `NSTrackingArea`) fires spurious `mouseExited` events at intermediate window sizes while AppKit animates the panel frame. Without a grace window, this creates an expand/collapse loop: hover → expand starts → transient exit → collapse starts → re-enter → loop. Fix: `NotchWindowController.handleHoverChange` defers the collapse by 0.25 s (> animation duration 0.22 s) via a cancellable `DispatchWorkItem`; `mouseEntered` cancels it immediately so real exit still collapses promptly. Do NOT remove or shorten the delay - the 0.25 s is load-bearing.
- **Panel frame animation - use `panel.animator().setFrame` not `setFrame(animate:)`:** `NSWindow.setFrame(_:display:animate:)` with `animate: true` uses `NSWindow.animationResizeTime`, ignoring any enclosing `NSAnimationContext`. To honor `NSAnimationContext` duration/curve (easeOut 0.22 s), use `panel.animator().setFrame(frame, display: true)` inside `NSAnimationContext.runAnimationGroup`. This routes through Core Animation and matches the SwiftUI content animation.
- **Top-anchor invariant:** `positionPanel` always computes `originY = screen.maxY - size.height - topGap`, so `originY + height = screen.maxY - topGap` is constant regardless of panel size. This pins the top edge under the notch and makes the panel grow downward. Any refactor that changes `originY` without preserving this invariant will break the anchor.


- **Spotify Web API source (`SpotifyWebController`):** Authorization Code + PKCE flow; client ID `5a058e7eb1b140a1a4b97bd801fc8734` (public - PKCE needs no secret), redirect URI `http://127.0.0.1:8888/callback`. Both in `SpotifyWebConfig` at the top of `SpotifyWebController.swift`. `SpotifyWebController` is a singleton (`static let shared`) so `MediaPane` can observe `authState` without threading `MediaController` through the settings view hierarchy. Tokens (`access_token`, `refresh_token`) live in Keychain service `notchmate.spotify.webapi` only - never plaintext, never logged. Token expiry timestamp (non-sensitive) is stored in UserDefaults key `spotifyWebTokenExpiry`. Auto-refresh: `withFreshToken` checks expiry before every API call and refreshes via `grant_type=refresh_token` if expired (60s grace margin). Polling runs at 2.5s and is stopped when the source is not `.spotifyWeb` (conserves Spotify API quota). Controls return 403 for non-Premium accounts - `premiumRequired` is set, controls dim, hint shown, reading still works. 429 stops polling for `Retry-After` seconds then resumes.
- **Spotify token storage - data-protection keychain (ROOT CAUSE + FIX for login-keychain password prompt):** `SpotifyKeychain` uses `kSecUseDataProtectionKeychain = true` on every SecItem call. Root cause of the repeated "notchmate wants to use your confidential information... enter login keychain password" prompt: the login keychain gates access via a per-item ACL bound to the app's binary hash. Every ad-hoc rebuild produces a new binary hash, so macOS sees a different app not on the ACL and prompts. "Always Allow" only persists for the exact signature - the next rebuild prompts again. The data-protection keychain gates access by entitlement (`keychain-access-groups`) rather than binary hash; the entitlement is stable across ad-hoc rebuilds. Required entitlement: `keychain-access-groups = ["com.notchmate.app"]` in `notchmate.entitlements` (literal string, NOT the Xcode `$(AppIdentifierPrefix)com.notchmate.app` variable - the headless `codesign` command doesn't expand Xcode build variables). Migration: `SpotifyKeychain.migrateLoginKeychainIfNeeded()` is called from `SpotifyWebController.start()`; it uses `kSecUseAuthenticationUIFail` to attempt a no-UI read of any token in the old login-keychain location and moves it to the new location, then deletes the orphan. Developer ID upgrade path: when switching from ad-hoc to Developer ID signing, update the `keychain-access-groups` entitlement value from `com.notchmate.app` to `TEAMID.com.notchmate.app` - the user will need to reconnect Spotify once (expected, new identity). The login-keychain ACL then sticks permanently and eliminates the per-rebuild prompt.
- **SpotifyWebController loopback listener:** NWListener on TCP 127.0.0.1:8888. Receives one HTTP GET `/callback?code=...`, sends a styled HTML response, then cancels itself. A 5-minute timeout (DispatchQueue.global asyncAfter) aborts the flow and resets `authState` to `.disconnected` if the browser is closed without completing login. Port 8888 is registered in the Spotify app dashboard; if changed, update both `SpotifyWebConfig.callbackPort` and the Spotify app's redirect URI list.
- **MediaSource.spotifyWeb and MediaController:** `MediaSource` now has three cases: `.spotify`, `.spotifyWeb`, `.nowPlaying`. `MediaController` holds `let spotifyWeb = SpotifyWebController.shared`. In `rebind()`, only the active source starts Web API polling (`spotifyWeb.startPolling()` when `.spotifyWeb`, `spotifyWeb.stopPolling()` otherwise). `LyricsController` uses `media.spotifyWeb.estimatedProgressSeconds()` for position when `.spotifyWeb` is the active source - a pure in-process computation, no network or blocking call.
- AppleScript MUST guard every call with `if application "Spotify" is running` - a bare `tell application "Spotify"` auto-launches Spotify.
The guard also keeps a closed Spotify silent (idle state, no error spam).
- `NSAppleScript` runs on a dedicated serial queue (not main); results published back to main.
Artwork is fetched from `artwork url` and cached per track id.
- **Apple Events on macOS 26 require hardened runtime + entitlement (ROOT CAUSE OF "prompt never fires"):** On macOS 26 (26.5.1, confirmed 2026-06-25), the OS blocks outgoing Apple Events for apps without hardened runtime (`--options runtime`) AND the `com.apple.security.automation.apple-events` entitlement. Without both:
  - The app never appears in System Settings > Privacy > Automation (TCC never records it).
  - No prompt fires.
  - Every Apple Event silently returns -1743 (`errAEEventNotPermitted`).

- **MediaRemote (private framework) on macOS 26 - restricted:** `MRMediaRemoteGetNowPlayingInfo` (`Media/MediaController.swift`, `SystemNowPlayingController`) is loaded at runtime via `dlopen`/`dlsym` from `/System/Library/PrivateFrameworks/MediaRemote.framework/MediaRemote`. On macOS 26 (26.5.1, confirmed 2026-06-25), the binary lives in the dyld shared cache (not on disk) but `dlopen` of the framework path still resolves through the shared cache - the function pointer is obtained successfully.
However, `MRMediaRemoteGetNowPlayingInfo` returns nil to its callback for apps without the private `com.apple.mediaremote` entitlement (reserved for first-party Apple apps). This means "Now Playing (any app)" will consistently show "Nothing playing" on macOS 15.4+ and macOS 26, even when music IS actively playing from another app.
The `SystemNowPlayingController` sets `unavailable = true` only when `dlopen`/`dlsym` completely fails (framework truly absent from shared cache). A nil callback result is treated as "nothing playing" - indistinguishable from silence.
`MRMediaRemoteSendCommand` (transport for the system NP source) is similarly a silent no-op on macOS 26 without entitlements.
**Consequence:** "Now Playing (any app)" is effectively non-functional on macOS 26. `MediaPane` surfaces a platform-version warning. Spotify source remains fully functional and is the recommended default.
- **MediaController and media source switching:** `NotchWindowController` owns one `MediaController` which wraps both `SpotifyController` and `SystemNowPlayingController`. - **Lyrics - LRCLIB source and caching:** `LyricsController` fetches from `https://lrclib.net/api/get` with `artist_name`, `track_name`, `album_name` query params. No API key required. 404 or missing both `syncedLyrics`/`plainLyrics` fields = `.noMatch`. Results are cached in-memory per `trackID` for the app session (no disk cache), so a skipped/revisited track does not re-fetch. - **Stopping a Claude session is the only destructive in-app action:** `ClaudeSessionsController.stop(_:)` sends `SIGTERM` (not `SIGKILL`) to exactly `pid_t(session.pid)` - the PID the hook recorded for that `claude` process - and never iterates or sweeps. The `ClaudeSessionsPane` always confirms via `.confirmationDialog` first, and maps `EPERM`/`ESRCH`/other errno to a clear alert. `kill` of another user's PID returns `EPERM` and surfaces as "Permission denied", never a crash.
- **Editing `~/.claude/settings.json` is sensitive - parse-merge-write only (`ClaudeHookInstaller`).** Never overwrite the file blindly. Load it as a dict, mutate only the `hooks` sub-tree, preserve every other key and the user's own hook entries, and write a `settings.json.notchmate-backup` first. A present-but-unparseable file MUST abort (`malformedSettings`), never clobber. Our entries are identified solely by the `notchmate-hook` substring in their command, so enable is idempotent (drop-then-add ours) and disable removes only ours (and prunes any now-empty event arrays / the `hooks` key). Validate changes via the `#if DEBUG runSelfCheck()`.

- **HUDController - CoreAudio volume listener:** uses `AudioObjectAddPropertyListenerBlock` (not the `AudioObjectPropertyListenerProc` C callback API) so the listener can be a Swift closure stored as `AudioObjectPropertyListenerBlock` and passed to both `AudioObjectAddPropertyListenerBlock` and `AudioObjectRemovePropertyListenerBlock` by identity. - **HUDController - brightness polling vs notification:** no public or private CoreDisplay/DisplayServices notification exists that reliably fires for brightness changes on Apple Silicon + macOS 26. - **Duck sprite sheets - format and adding characters:** Sheets are `192×128` RGBA PNG, a `6 columns × 4 rows` grid of `32×32` frames (24 slots total). Row layout (adjust in `DuckyAnimation` switch if a future sheet differs):

  - Row 1: `walk_normal` - 6 frames (thinking mood / Claude sessions active)
